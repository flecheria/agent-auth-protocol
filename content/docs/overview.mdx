Everything we've built for auth on the web assumes two kinds of actors: a human user and a static application, with predefined scopes and known execution paths.

Agents fit neither role. They act on behalf of a user or entirely on their own, call external services, discover tools at runtime, need one capability now and a different one later, and often run long after the human who started them has moved on.

These problems share a single root cause: **existing auth has no concept of "agent."** The server sees a credential, and credentials identify resources — not the specific actor using them. Without per-agent identity, the server cannot tell which agent made a request, cannot scope one differently from another, and cannot revoke one without affecting every other. There is no lifecycle, no state transitions, no way to expire an agent that should have stopped running hours ago.

Agent Auth makes the runtime agent a first-class principal. Each agent is registered with its own identity, granted specific capabilities, and governed by a lifecycle the server controls. The server sees exactly which agent is acting, what it is authorized to do, and can terminate one without affecting anything else.

## Delegated Agents

When an agent acts on behalf of a user, it typically uses the user's token or a shared client credential. That collapses the runtime agent into someone else's identity:

- **No visibility:** The server can't tell which agent made a request. There's no audit trail per agent — every call looks like it came from the user or at best the client application.
- **No scoping:** You can't scope each agent separately. All agents sharing the same credential get the same permissions — you can't give one agent narrower access than another.
- **No isolation:** You can't revoke one agent without revoking all of them. A compromised agent means rotating credentials for everything.

Agent Auth gives each agent its own keypair and its own registration. Every request is signed by the specific agent that made it. The server verifies it cryptographically and sees both the user and the exact agent acting for them. Instead of broad scopes like "write access to accounts," the server grants named capabilities with per-field constraints — `transfer_domestic` capped at $1,000 to a specific account. Each grant is scoped to the individual agent and independently revocable.

## Autonomous Agents

When an agent needs to act on its own — without a user in the loop — there is no identity model for it. Existing auth tightly couples the human user and the application. The agent is forced to pretend to be a human: opening a browser, solving a CAPTCHA, and clicking through a signup flow just to use a service.

Agent Auth gives each autonomous agent its own identity. No user session, no impersonation. The agent registers directly with its own keypair, authenticates with signed JWTs, and the server attributes every action to the exact agent instance.

## Discovery

There's no standard way for a service to advertise that it supports agents. Every new integration requires hardcoded configuration or a human pointing the agent to the right endpoint.

Agent Auth defines a well-known endpoint and a directory protocol so agents can find and connect to services automatically:

- **By URL** — the agent fetches `/.well-known/agent-configuration` from a domain and gets the full configuration.
- **By intent** — the agent describes what it wants to do in natural language, and a directory returns matching services.

## What the specification covers

The full [specification](/specification) defines the complete protocol across these areas:

1. **Identity** — how agents and hosts establish cryptographic identity with Ed25519 keypairs
2. **Registration** — how agents register under a host on a server and receive capability grants
3. **Authentication** — how agents authenticate requests with short-lived signed JWTs
4. **Capabilities** — how servers define, grant, constrain, and revoke capabilities per agent
5. **Approval** — how user consent works for registration, linking, and capability escalation
6. **Discovery** — how agents find services via well-known endpoints and directory search
7. **Lifecycle** — how agents transition between active, suspended, expired, and revoked states
